# Repte 3. Framework, arquitectura i persistència

## Finalitat del repte

Ací és on entra el contrast de frameworks del curs.

El mateix projecte de `R2` continua ara amb:

- `Laravel`
- `Symfony`
- `NestJS`

El producte i l’encàrrec funcional no canvien. El que canvia és la manera d’organitzar-lo, fer-lo més mantenible i consolidar la persistència.

`FastAPI` només podria aparéixer com a via avançada o excepcional, no com a itinerari base general.

## Què has de consolidar

- una estructura més clara del backend
- separació de responsabilitats
- persistència segura i explicable
- la funcionalitat protegida que ja venia de `R2`
- validacions, errors i proves mínimes

## Què no és suficient

- reescriure de zero i perdre el que ja funcionava
- fer una arquitectura decorativa
- connectar una base de dades sense un flux complet
- parlar només del framework sense demostrar el producte

## Evidència mínima

- un cas d’ús real migrat o consolidat de punta a punta
- persistència funcional i verificable
- commits amb migració recognoscible
- `README` i decisions tècniques actualitzades
- prova que la funcionalitat principal continua viva

## Cooperació en este tram

És el moment natural per a les parelles de contrast tècnic.

Podeu compartir context, checkpoints i comparació de solucions, però:

- cada persona manté el seu repositori
- cada persona deixa les seues evidències
- cada persona defensa el seu propi codi

Els trios avançats només tenen sentit com a opció puntual.

## Connexió amb el següent pas

Si `R3` queda ben tancat, en `R4` podràs exposar una API real del producte sense penjar-la sobre una base dèbil.
