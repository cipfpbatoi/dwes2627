# Brief Gamma. R3. Framework, arquitectura i persistència real

## Objectiu

Crear una presentació de llançament del `Repte 3` per explicar la reconstrucció d'una part del projecte en un framework, amb arquitectura clara, persistència real, dades inicials reproduïbles, dos fluxos funcionals, autenticació, autorització i estabilització tècnica.

## Audiència

Alumnat que ve de `R2` amb una funcionalitat de servidor en PHP base i necessita fer el salt a una estructura professional.

## Missatge clau

`R3` no és copiar el projecte dins d'un framework. És reconstruir una part valuosa amb millor arquitectura, dades persistents, fluxos verificables i criteris comuns que es poden defensar en Laravel, Symfony, NestJS o un stack autoritzat.

## Estructura recomanada

1. **Portada**
   - `R3. Framework, arquitectura i persistència real`.
   - Subtítol: del flux funcional a un producte més mantenible.
2. **Punt de partida**
   - `R2` deixa un flux backend funcional.
   - `R3` selecciona què val la pena reconstruir.
3. **Què canvia amb un framework**
   - Rutes.
   - Controladors.
   - Vistes o respostes.
   - Models o serveis.
   - Configuració.
   - Proves i entorn.
4. **Límit del repte**
   - No cal migrar tota l'aplicació.
   - Cal reconstruir una part amb sentit i evidència clara.
5. **Objectiu del repte**
   - Arrancar un projecte en framework.
   - Crear persistència real.
   - Carregar dades inicials.
   - Implementar dos fluxos funcionals.
   - Protegir una acció.
   - Deixar proves i README.
6. **Stacks possibles**
   - Laravel.
   - Symfony.
   - NestJS.
   - Altres opcions només si estan justificades i autoritzades.
   - El criteri important és poder explicar equivalències i evidències.
7. **Microreptes**
   - `R3M1`: arrencada del nou projecte.
   - `R3M2`: persistència mínima i dades inicials reproduïbles.
   - Taller de portabilitat: entendre equivalències entre frameworks.
   - `R3M3`: primer cas d'ús complet heretat de `R2`.
   - `R3M4`: segon cas d'ús o ampliació funcional.
   - `R3M5`: autenticació, autorització i middleware.
   - `R3M6`: qualitat i estabilització.
   - `R3M7`: tancament tècnic.
8. **Evidències principals**
   - Projecte que arranca.
   - `.env.example` o configuració reproduïble.
   - Migracions o mecanisme equivalent.
   - Seeders o càrrega inicial equivalent.
   - Dos fluxos funcionals.
   - Acció protegida en ruta o controlador.
   - Proves o checklist de regressió.
   - README de tancament.
9. **Errors habituals**
   - Crear només l'esquelet del framework.
   - Carregar dades a mà.
   - Fer pantalles sense flux.
   - Protegir només el botó i no la ruta.
   - No documentar l'arrencada.
   - Dependre d'un entorn local invisible.
10. **Connexió amb R4**
    - Quan el producte té dades i fluxos estables, ja pot exposar una part com a API.
11. **Checklist final**
    - El framework arranca.
    - Les dades es poden reiniciar i carregar.
    - Hi ha dos fluxos verificables.
    - L'acció protegida no és decorativa.
    - El README permet repetir la demo.
    - Les decisions tècniques es poden defensar.

## Format recomanat

- 10-12 diapositives.
- Incloure un esquema visual: `R2 funcional -> R3 framework -> BBDD -> fluxos -> R4 API`.
- Usar exemples de producte reals.
- Incloure una taula d'equivalències entre frameworks sense fer tutorial d'un stack concret.
- Tancar amb checklist final del repte.

## No incloure

- Tutorial exhaustiu de Laravel, Symfony o NestJS.
- Comandos llargs d'instal·lació.
- Migració completa de tota l'aplicació.
- API com a centre del repte.
