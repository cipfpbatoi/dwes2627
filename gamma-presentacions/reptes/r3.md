## Contingut que ha d'incloure

- Context: `R2` deixa una funcionalitat de servidor; `R3` la reconstrueix amb framework i persistència real.
- Límit clau: no migrar tota l'aplicació.
- Stacks base: `Laravel`, `Symfony`, `NestJS`.
- Evidències globals: Docker, `.env.example`, rutes, controladors, vistes o respostes, BBDD, migracions o equivalent, dades inicials reproduïbles, dos fluxos, acció protegida, proves i README.
- Seqüència:
  - `R3M1`: arrencada del nou projecte.
  - `R3M2`: persistència mínima amb migracions o equivalent i dades inicials reproduïbles.
  - `R3S2B`: taller de portabilitat entre frameworks abans del primer flux complet.
  - `R3M3`: primer cas d'ús complet heretat de `R2`.
  - `R3M4`: segon cas d'ús o ampliació funcional.
  - `R3M5`: autenticació, autorització i middleware.
  - `R3M6`: qualitat i estabilització.
  - `R3M7`: tancament tècnic.
- Criteris principals: `RA5`, `RA6` i `RA8`.
- Errors habituals:
  - crear només un esquelet de framework;
  - carregar dades a mà;
  - fer una pantalla sense flux;
  - no tindre cas server-rendered;
  - protegir només la vista i no la ruta;
  - no deixar README reproduïble.
- Connexió amb `R4`: una base persistent i mantenible pot exposar-se com a API.

## Criteris de disseny

- Presentació de 10-12 diapositives.
- Incloure un esquema visual: `R2 funcional -> R3 framework -> BBDD -> dos fluxos -> R4 API`.
- Usar exemples de producte reals, no exemples genèrics.
- Tancar amb checklist de superació del repte.

## Límit

No convertir la presentació en tutorial complet de Laravel, Symfony o NestJS. La presentació ha d'explicar criteris comuns i evidències, no comandos exhaustius.
