# Brief Gamma. R4. API pròpia, consum i documentació

## Objectiu

Crear una presentació de llançament del `Repte 4` per explicar com convertir una funcionalitat real del projecte en una API usable, protegida quan cal, consumida des d'un client o eina externa i documentada de forma executable.

## Audiència

Alumnat que acaba `R3` amb un projecte en framework, persistència mínima i almenys un flux funcional.

## Missatge clau

`R4` no és afegir rutes soltes que tornen `JSON`. És publicar un servei sobre una funcionalitat real, amb contracte clar, resposta verificable, control d'accés, consum real i documentació que una altra persona pot repetir.

## Estructura recomanada

1. **Portada**
   - `R4. API pròpia, consum i documentació`.
   - Subtítol: del producte intern al servei consumible.
2. **Punt de partida**
   - Projecte de `R3`.
   - Dades persistents.
   - Almenys un flux funcional que té valor.
3. **Què canvia en R4**
   - Una pantalla és per a persones.
   - Una API és per a altres programes.
   - El contracte substitueix la intuïció.
4. **Objectiu del repte**
   - Definir un contracte.
   - Implementar un endpoint funcional.
   - Protegir l'accés si correspon.
   - Consumir l'API.
   - Documentar i provar.
5. **Microreptes**
   - `R4M1`: contracte del servei.
   - `R4M2`: primer endpoint funcional.
   - `R4M3`: protecció de l'API.
   - `R4M4`: consum real del servei.
   - `R4M5`: documentació i proves.
6. **Contracte mínim**
   - Mètode HTTP.
   - URL.
   - Paràmetres o body.
   - Capçaleres.
   - Resposta correcta.
   - Respostes d'error.
   - Exemple reproduïble.
7. **Evidències principals**
   - Contracte escrit.
   - Endpoint amb dades reals.
   - Cas correcte.
   - Cas d'error.
   - Cas autoritzat i denegat si hi ha protecció.
   - Consum des d'una eina, script o client.
   - Documentació executable.
8. **Què no és suficient**
   - `JSON` estàtic.
   - Rutes decoratives.
   - Documentació que no es pot executar.
   - Consum que només funciona en una màquina.
   - Protecció que no es prova.
9. **Connexió amb R5**
   - Una API ben documentada prepara la integració.
   - Un contracte fràgil dificulta qualsevol sistema híbrid.
10. **Checklist final**
    - El contracte és clar.
    - L'endpoint usa dades del producte.
    - Els errors estan definits.
    - La protecció està provada si aplica.
    - Hi ha un consumidor real.
    - La documentació permet repetir les peticions.

## Format recomanat

- 10-12 diapositives.
- Diagrama: funcionalitat interna -> contracte -> endpoint -> consumidor.
- Taula de microreptes amb evidència principal.
- Exemple curt de request/response.
- Flux de prova amb cas correcte, cas d'error i cas denegat.
- Tancar amb checklist final del repte.

## No incloure

- Teoria extensa de protocols.
- OAuth o JWT avançat com a requisit universal.
- Integracions externes pròpies de `R5`.
- Debat genèric sobre REST sense connexió amb el producte.
