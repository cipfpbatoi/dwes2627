# Brief Gamma. Repte 4. API i consum

## Objectiu

Crear una presentació de llançament del `Repte 4`, centrada en convertir una funcionalitat real del projecte en una `API` usable, protegida, consumida i documentada.

## Audiència

Alumnat de DWES que acaba `R3` amb un projecte en framework, persistència mínima i almenys un flux funcional.

## Missatge clau

`R4` no és fer rutes soltes. És publicar un servei web sobre una funcionalitat real, amb contracte clar, resposta verificable, control d'accés quan cal, consum real i documentació executable.

## Estructura proposada

1. Què canvia en `R4`
   - De backend intern a servei reusable.
   - De pantalla/flux a contracte consumible.
2. Punt de partida
   - Funcionalitat real de `R3`.
   - Dades persistents i flux defensable.
3. Microreptes del repte
   - `R4M1`: contracte del servei.
   - `R4M2`: primer endpoint funcional.
   - `R4M3`: protecció de l'API.
   - `R4M4`: consum real del servei.
   - `R4M5`: documentació i proves.
4. Evidències mínimes
   - Contracte, endpoint, cas autoritzat/denegat, consumidor, documentació i proves.
5. Què no és suficient
   - Rutes decoratives, JSON estàtic, documentació ornamental o consum no reproduïble.
6. Relació amb `R5`
   - Una API clara pot alimentar integració híbrida.
   - Un contracte fràgil trenca el pas següent.

## To

Pràctic, orientat a producte i verificació. Evita una presentació teòrica genèrica sobre REST.

## Elements visuals suggerits

- Diagrama: funcionalitat interna -> contracte -> endpoint -> consumidor.
- Taula de microreptes amb evidència principal.
- Exemple de contracte request/response.
- Flux de prova: cas correcte, cas d'error, cas denegat.

## No incloure

- Teoria extensa de protocols.
- OAuth/JWT avançat com a requisit.
- Integracions externes de `R5` com a centre del repte.
