# Brief Gamma. R1. Kickoff backend i base executable

## Objectiu

Crear una presentació de llançament del `Repte 1` per situar l'alumnat en el treball professional de backend: entendre el model client-servidor, triar una base tecnològica raonable i deixar un primer projecte executable, documentat i reproduïble.

## Audiència

Alumnat que comença DWES i necessita passar d'una idea general de "fer webs" a una primera evidència backend verificable.

## Missatge clau

`R1` no és instal·lar ferramentes sense criteri. És demostrar que entens què fa el backend, com respon un servidor i com deixes una base executable que una altra persona pot arrancar i revisar.

## Estructura recomanada

1. **Portada**
   - `R1. Kickoff backend i base executable`.
   - Subtítol: del navegador a un primer servidor verificable.
2. **Què canvia en DWES**
   - El navegador demana.
   - El servidor processa.
   - El backend respon.
   - L'evidència no és una captura bonica, és un comportament reproduïble.
3. **El problema professional**
   - Un equip necessita una base comuna.
   - El projecte ha d'arrancar en local.
   - Les versions i dependències han d'estar documentades.
4. **Objectiu del repte**
   - Construir una base mínima de backend executable.
   - Entendre les peces del stack.
   - Deixar una comprovació clara de funcionament.
5. **Mapa conceptual**
   - Client.
   - Servidor web.
   - Runtime o llenguatge.
   - Codi del projecte.
   - Dependències.
   - Entorn local o contenidor.
6. **Seqüència de treball**
   - Entendre backend i client-servidor.
   - Analitzar stacks possibles.
   - Preparar l'entorn.
   - Crear una ruta o resposta mínima.
   - Documentar l'arrencada.
   - Revisar que és reproduïble.
7. **Producte final esperat**
   - Repositori ordenat.
   - Projecte que arranca.
   - Ruta inicial o `health check`.
   - README amb passos d'arrencada.
   - Evidència que una altra persona pot comprovar.
8. **Evidències principals**
   - Comanda d'arrencada.
   - URL de comprovació.
   - Captura o registre de resposta.
   - Versions utilitzades.
   - Explicació breu del stack triat.
9. **Errors habituals**
   - Confondre frontend amb backend.
   - Dir "funciona en el meu ordinador" sense README.
   - No fixar versions.
   - Pujar secrets o fitxers locals innecessaris.
   - Fer una estructura que ningú pot reproduir.
10. **Connexió amb R2**
    - En `R1` el servidor respon.
    - En `R2` el servidor rebrà dades, validarà, decidirà i mantindrà estat.
11. **Checklist final**
    - Puc explicar què fa el backend en el meu projecte.
    - El projecte arranca amb una comanda clara.
    - Hi ha una URL de comprovació.
    - El README permet repetir l'arrencada.
    - El stack triat està justificat.

## Format recomanat

- 10-12 diapositives.
- To visual net, pràctic i orientat a producte.
- Incloure un diagrama simple navegador -> servidor -> resposta.
- Usar exemples propers al projecte base de l'alumnat.
- Tancar amb checklist final del repte.

## No incloure

- Instal·lació detallada pas a pas.
- Teoria extensa de xarxes.
- Comparativa profunda de frameworks.
- Contingut propi de `R2`: formularis, sessions, autenticació o BBDD.
