# Gamma brief - R2M6. Mini API d'autenticació per a client

## RA avaluat

- **RA avaluat**: `RA4`
- **CA avaluats**: `RA4.d`, `RA4.e`
- **RA de context**: `pont DWEC`

Este microrepte genera nota només per al RA avaluat. Els RA de context poden aparéixer en el producte, però no es ponderen en esta rúbrica.


## Objectiu de la presentació

Presentar el microrepte `R2M6` com un pont directe entre DWES i DWEC: exposar l'autenticació ja creada en `R2M5` com una mini API consumible des d'un client.

## Públic

Alumnat que ja ha implementat registre, hash, login, sessió i una operació protegida en `R2M5`.

## Missatge central

Ara el backend no només respon al navegador amb HTML: també pot respondre a un altre programa amb `JSON`.

## Estructura suggerida

1. **Per què ara una mini API?**
   - DWEC necessita autenticar-se contra un backend.
   - R4 queda massa lluny.
   - Ací fem només el mínim consumible.

2. **Què reutilitzem**
   - Usuaris de `R2M5`.
   - Validació de contrasenya.
   - Operació protegida.
   - Regles bàsiques d'accés.

3. **Contracte mínim**
   - `POST /api/login`.
   - Entrada amb usuari/contrasenya.
   - Resposta `JSON`.
   - Token simple.
   - `GET /api/me` o equivalent.
   - Header `Authorization`.

4. **Respostes clares**
   - `200` en cas correcte.
   - `401` en cas incorrecte.
   - No retornar contrasenyes ni hashes.
   - Missatge d'error interpretable.

5. **Prova externa**
   - `curl`.
   - Postman/Insomnia.
   - `fetch` des de client.
   - Cas correcte i cas incorrecte.

6. **Què no és encara**
   - API completa.
   - CRUD.
   - OpenAPI.
   - OAuth.
   - JWT professional.
   - Refresh tokens.
   - Rols complexos.

7. **Pont cap a R4**
   - En R4 es formalitzarà contracte, documentació i protecció.
   - Ara només cal una porta mínima i estable per a DWEC.

## To visual

## Frase final

`R2M6` no és una API completa: és el primer contracte d'autenticació perquè el client puga parlar amb el backend.
